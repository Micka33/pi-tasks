# Plan d’implémentation — pi-tasks

## Statut

Implémentation initiale réalisée : package Pi, serveur MCP stdio, cœur SQLite partagé, refresh de claim, visibilité privée stricte avec bypass, soft-delete, et tests automatisés.


## Décisions clarifiées

- Livrer **les deux** :
  1. un **package Pi** avec extension TypeScript ;
  2. un **vrai serveur MCP** exposant les mêmes outils.
- Stockage : **SQLite**, pas JSON.
- TTL par défaut des claims : **2 heures**.
- Ajouter une capacité de **refresh de claim** sans modifier `started_at`, afin de calculer la durée réelle de la tâche.
- `agent_id` côté Pi : dérivé automatiquement de la **session Pi**.
- `visibility = private` : appliqué strictement, avec bypass seulement après confirmation utilisateur explicite et justification.
- `task_delete` : **soft-delete**, pas suppression physique.
- `task_update(status = in_progress)` : interdit. Le passage en `in_progress` doit passer exclusivement par `task_claim_next`.

```json

  "name": "pi-tasks",
  "keywords": ["pi-package"],
  "pi": {
    "extensions": ["./src/pi/index.ts"]

  "bin": {
```

Installation Pi :

```bash
pi install git:git@github.com:Micka33/pi-tasks.git
```

Test local Pi :

```bash

```

## Serveur MCP

Framework proposé : **SDK TypeScript officiel** `@modelcontextprotocol/sdk`.

Déploiement proposé pour commencer : **local stdio**, car le serveur doit accéder à une base SQLite locale/partagée.

Commande cible :

```bash
PI_TASKS_DB_PATH=/path/to/tasks.sqlite pi-tasks-mcp
```

À documenter pour Claude Desktop / Claude Code : configuration MCP locale pointant sur ce binaire.

Si tu veux distribuer largement ce serveur à des utilisateurs qui n’ont pas Node configuré, une phase ultérieure pourra packager le serveur en **MCPB**.

## Base SQLite

Chemin proposé par défaut :

```text
.pi/pi-tasks/tasks.sqlite
```

Override :

```bash
PI_TASKS_DB_PATH=/chemin/partage/tasks.sqlite
```

Pragmas proposés :

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;
```

SQLite gère le verrouillage multi-process. Les mutations critiques seront faites dans des transactions `BEGIN IMMEDIATE` pour rendre `task_claim_next` atomique.

## Schéma SQLite proposé

### `task_lists`

```sql
CREATE TABLE task_lists (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  scope_type TEXT NOT NULL CHECK (scope_type IN ('workspace', 'thread', 'agent', 'global', 'custom')),
  scope_key TEXT NOT NULL,
  visibility TEXT NOT NULL CHECK (visibility IN ('private', 'shared')),
  owner_agent_id TEXT,
  created_by_agent_id TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  deleted_at TEXT
);
```

### `tasks`

```sql
CREATE TABLE tasks (
  id TEXT PRIMARY KEY,
  list_id TEXT NOT NULL REFERENCES task_lists(id),
  position INTEGER NOT NULL,
  title TEXT NOT NULL,
  description TEXT,
  notes TEXT,
  status TEXT NOT NULL CHECK (status IN ('todo', 'in_progress', 'blocked', 'done', 'canceled')),
  assigned_to_agent_id TEXT,
  claimed_by_agent_id TEXT,
  claim_expires_at TEXT,
  result TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  started_at TEXT,
  completed_at TEXT,
  deleted_at TEXT
);
```

Indexes :

```sql
CREATE INDEX idx_task_lists_scope ON task_lists(scope_type, scope_key, visibility);
CREATE INDEX idx_tasks_list_position ON tasks(list_id, position);
CREATE INDEX idx_tasks_claim ON tasks(status, claim_expires_at);
CREATE INDEX idx_tasks_assignment ON tasks(list_id, assigned_to_agent_id, status);
```

## Soft-delete

`task_delete` ne supprimera pas physiquement la tâche. Il définira :

```text
deleted_at = now
updated_at = now
```

Par défaut, les outils ne retourneront pas les tâches supprimées. `task_list_get` et `task_lists_find` pourront avoir une option `include_deleted`.

Même logique possible pour les listes si on ajoute ensuite `task_list_delete`.

## Dérivation `agent_id`

### Dans Pi

Dériver automatiquement à partir du fichier de session Pi :

```text
agent_id = "pi-session:" + sha256(sessionFile).slice(0, 16)
```

- Stable si la session Pi est reprise.
- Ne révèle pas le chemin local complet.
- Si Pi tourne avec `--no-session`, fallback proposé : `PI_TASKS_AGENT_ID`, sinon `hostname + pid`, avec avertissement car non stable.

### Dans MCP

MCP n’a pas de session Pi. Proposition :

1. utiliser `PI_TASKS_AGENT_ID` si défini ;
2. sinon dériver un identifiant de session MCP/process ;
3. documenter que, pour un comportement stable entre redémarrages MCP, `PI_TASKS_AGENT_ID` est recommandé.

Ce point reste à valider.

## Règles de visibilité privée

Pour toute opération sur une liste privée :

- autoriser si `resolved_agent_id === owner_agent_id` ;
- autoriser si `resolved_agent_id === created_by_agent_id` quand aucun owner n’est défini ;
- sinon refuser par défaut.

Bypass :

- Côté Pi interactif : afficher une confirmation via `ctx.ui.confirm()` avec explication claire : liste privée, propriétaire, outil demandé, raison du bypass.
- Côté Pi non interactif : refuser et retourner une erreur exploitable.
- Côté MCP : utiliser l’**elicitation MCP** si le client la supporte ; sinon retourner une erreur demandant une confirmation explicite de l’utilisateur via un paramètre de retry contrôlé.

Le bypass devra être journalisé dans le résultat de l’outil et dans les détails retournés.

## Outils exposés dans Pi et MCP

Les mêmes noms seront exposés dans les deux surfaces.

### `task_list_create`
Créer une liste. `id` optionnel ; si absent, `crypto.randomUUID()`.

### `task_lists_find`
Retrouver les listes selon scope, visibilité, owner, créateur, nom. Exclut les listes soft-deleted par défaut.

### `task_list_get`
Lire une liste et ses tâches triées par `position`. Exclut les tâches soft-deleted par défaut.

### `task_create`
Ajouter une tâche unique.

### `task_add_many`
Ajouter plusieurs tâches dans une transaction unique.

### `task_claim_next`
Seule méthode normale pour prendre une tâche :

1. transaction `BEGIN IMMEDIATE` ;
2. libération optionnelle des claims expirés ;
3. recherche de la première tâche `todo`, non supprimée, triée par `position` ;
4. respect de `assigned_to_agent_id` ;
5. passage à `in_progress` ;
6. définition de `claimed_by_agent_id`, `claim_expires_at = now + 2h`, `started_at` si vide, `updated_at` ;
7. retour de la tâche ou `null`.

### `task_claim_refresh`
Nouvel outil à ajouter pour ton besoin de heartbeat :

- vérifie que la tâche est `in_progress` ;
- vérifie que `claimed_by_agent_id === agent_id` ;
- met à jour `claim_expires_at = now + ttl` ;
- met à jour `updated_at` ;
- **ne modifie jamais `started_at`**.

### `task_update`
Modifier titre, description, notes, assignment, result ou statut, sauf `in_progress`.

Règles statut :

- `done` / `canceled` : définit `completed_at`, vide claim et expiration.
- `blocked` : vide claim et expiration, conserve notes/résultat si fourni.
- `todo` : vide claim et expiration ; `started_at` reste conservé pour audit.
- `in_progress` : rejet systématique avec message indiquant d’utiliser `task_claim_next`.

### `task_reorder`
Réordonner les tâches et renuméroter `position`.

### `task_release_expired_claims`
Libérer les claims expirés en repassant les tâches concernées à `todo`, sans modifier `started_at`.

### `task_delete`
Soft-delete de tâche.

## Guidelines injectées dans Pi et descriptions MCP

Les descriptions d’outils rappelleront :

- `task_claim_next` est l’unique voie normale vers `in_progress` ;
- ne jamais prendre une tâche depuis `task_list_get` pour la passer manuellement en `in_progress` ;
- utiliser `task_claim_refresh` pour prolonger un claim long ;
- toujours terminer par `task_update` (`done`, `blocked`, `todo`, `canceled`) ;
- respecter strictement les listes privées ;
- utiliser `blocked` si une information manque.

## Commandes slash Pi optionnelles

- `/task-lists` : afficher les listes visibles par la session Pi courante.
- `/tasks <list_id>` : afficher une liste compacte.
- `/task-store` : afficher le chemin SQLite utilisé.
- `/task-agent` : afficher l’`agent_id` dérivé de la session Pi.

## Tests prioritaires

1. Migrations et création DB.
2. Création de listes et tâches.
3. Tri par `position`.
4. `task_claim_next` retourne la bonne tâche.
5. Deux agents/process ne peuvent pas réclamer la même tâche.
6. Une tâche assignée ne peut être réclamée que par le bon agent.
7. TTL par défaut = 2h.
8. `task_claim_refresh` prolonge `claim_expires_at` sans changer `started_at`.
9. `task_release_expired_claims` remet à `todo` sans effacer l’audit.
10. `task_update(status = in_progress)` est rejeté.
11. `done`, `blocked`, `canceled`, `todo` nettoient correctement le claim.
12. Visibilité privée appliquée strictement.
13. Bypass privé journalisé.
14. `task_delete` fait un soft-delete et les reads l’excluent par défaut.
15. Même DB SQLite + même `list_id` = liste partagée entre extension Pi et serveur MCP.
